# Unscramble Game

## Описание проекта
**Unscramble Game** — это образовательное Android-приложение, представляющее собой игру в слова. Цель игры — угадать исходное слово из набора перемешанных букв. Приложение разработано с использованием современных архитектурных компонентов Android Jetpack для демонстрации правильных подходов к управлению состоянием UI и сохранению данных при изменениях конфигурации устройства (например, повороте экрана).

## Технологии
Проект реализован с использованием следующего стека технологий:
*   **Kotlin** — основной язык программирования.
*   **Jetpack Compose** — современный toolkit для создания нативных UI.
*   **ViewModel** — компонент архитектуры для хранения и управления данными интерфейса.
*   **StateFlow** — потоковый механизм для реактивного обновления UI при изменении состояния.
*   **Material Design 3** — для оформления элементов интерфейса.

## Функциональность приложения
*   **Генерация заданий:** Случайный выбор слова из списка и его перемешивание.
*   **Ввод ответа:** Поле ввода текста для попытки угадать слово.
*   **Проверка ответа:**
    *   При правильном ответе: начисление очков (20 баллов) и переход к следующему слову.
    *   При ошибке: отображение сообщения об ошибке (красная подсветка поля ввода).
*   **Подсказка/Пропуск:** Возможность пропустить текущее слово без начисления очков.
*   **Сохранение состояния:** Данные игры (счет, текущее слово, номер вопроса) не теряются при повороте экрана или смене темы.
*   **Завершение игры:** После отгадывания 10 слов отображается диалог с итоговым счетом и кнопкой «Играть снова».
*   **Unit-тесты:** Покрытие логики ViewModel тестами для проверки корректности начисления очков и обработки ошибок.

# Ответы на контрольные вопросы

### 1. Что такое ViewModel и зачем он нужен?
**ViewModel** — это класс-компаньон для UI-контроллера (Activity или Composable), предназначенный для хранения данных интерфейса и бизнес-логики.
*   **Какую проблему решает:** Главная проблема, которую решает ViewModel — это потеря данных при **изменении конфигурации** (configuration change), например, при повороте экрана. Обычные переменные внутри Composable-функций или Activity уничтожаются вместе с ними при пересоздании. ViewModel живет дольше, чем UI-компоненты, и сохраняет свое состояние пока активен процесс приложения.
*   **Без ViewModel:** При повороте экрана Activity уничтожается и создается заново. Все обычные переменные (например, `var score = 0`) сбрасываются к начальным значениям, и прогресс пользователя теряется.

### 2. В чём разница между MutableStateFlow и StateFlow?
*   **MutableStateFlow:** Это изменяемая версия потока. Позволяет как читать значение, так и обновлять его (свойство `value`). Используется внутри ViewModel для изменения состояния.
*   **StateFlow:** Это неизменяемая (read-only) версия потока. Позволяет только наблюдать за изменениями и читать текущее значение. Передается наружу (в UI) для отображения.
*   **Зачем два поля (`_uiState` и `uiState`):** Это реализация паттерна инкапсуляции. Приватное поле `_uiState` (MutableStateFlow) нужно, чтобы логика внутри ViewModel могла менять данные. Публичное поле `uiState` (StateFlow) отдается UI, чтобы интерфейс мог только следить за изменениями, но не мог случайно изменить состояние игры напрямую.
*   **Backing property pattern:** Это шаблон проектирования, при котором приватное изменяемое свойство («подложка») скрывается от внешнего мира, а публичное свойство предоставляет только доступ на чтение. Это защищает целостность данных.

### 3. Почему в ViewModel нельзя хранить Context или View?
*   **Причина:** ViewModel имеет жизненный цикл, который длиннее, чем у Activity или View. Activity может быть уничтожена и создана заново (при повороте экрана), а ViewModel останется жить.
*   **Последствия:** Если сохранить ссылку на Activity (Context) или View внутри ViewModel, то после уничтожения Activity сборщик мусора (Garbage Collector) не сможет удалить её из памяти, так как на неё всё ещё есть ссылка из живого ViewModel. Это приводит к **утечке памяти (Memory Leak)**, что вызывает падение производительности и краш приложения.

### 4. Чем StateFlow отличается от remember и rememberSaveable?
*   **remember:** Хранит состояние только пока Composable-функция находится в композиции. При пересоздании Activity (поворот экрана) данные теряются. Подходит для временного UI-состояния (например, открыто ли меню прямо сейчас).
*   **rememberSaveable:** Сохраняет данные в Bundle при уничтожении процесса. Переживает поворот экрана и даже смерть процесса. Однако подходит только для простых типов данных и требует сериализации. Логика часто остается привязанной к UI.
*   **StateFlow (в ViewModel):** Хранит данные в отдельном классе, переживает поворот экрана, но **не** переживает полную смерть процесса (если не использовать SavedStateHandle). Идеально подходит для хранения бизнес-логики и сложного состояния UI, обеспечивая разделение ответственности между логикой и отображением.

### 5. Что такое data class в Kotlin и зачем он используется для UI State?
*   **Определение:** `data class` — это специальный класс в Kotlin, предназначенный в первую очередь для хранения данных.
*   **Автоматически генерируемые методы:** Компилятор автоматически создает методы `equals()`, `hashCode()`, `toString()` и `copy()`.
*   **Зачем нужна функция `copy()`:** Она позволяет создать новую копию объекта, изменив только некоторые поля, оставив остальные без изменений. Это критически важно для работы с неизменяемыми (immutable) состояниями в StateFlow. Мы не меняем старый объект, а создаем новый с обновленными данными: `state.copy(score = 10)`.

### 6. Как работает collectAsState() в Compose?
*   **Механизм:** Функция `collectAsState()` преобразует холодный поток `StateFlow` в горячее состояние Compose (`State`). Она подписывается на поток и начинает собирать данные.
*   **Автообновление UI:** Когда значение внутри `StateFlow` меняется, `collectAsState()` уведомляет Compose о том, что данные изменились. Compose запускает механизм **рекомпозиции** — перерисовывает только те части экрана, которые зависят от изменившегося значения. Это происходит автоматически и эффективно.

### 7. Зачем разделять код на пакеты data и ui?
*   **Цель:** Разделение кода по уровням ответственности (Separation of Concerns).
*   **Принципы архитектуры:**
    *   **Модульность:** Легче найти нужный код (модели данных в `data`, логика отображения в `ui`).
    *   **Тестируемость:** Логику и модели данных можно тестировать независимо от UI.
    *   **Читаемость:** Новым разработчикам проще понять структуру проекта.
    *   Это поддерживает принципы чистой архитектуры и SOLID, предотвращая создание «божественных классов», где всё намешано в одном файле.

### 8. Что происходит с ViewModel, когда приложение полностью закрывается (process death)?
*   **Судьба данных:** ViewModel хранится в оперативной памяти. Когда операционная система убивает процесс приложения (например, из-за нехватки памяти или принудительного закрытия пользователем), вся память освобождается. **Данные в ViewModel теряются.**
*   **Альтернативы для долговременного хранения:** Чтобы данные пережили смерть процесса, необходимо использовать постоянные хранилища:
    *   **Room Database:** Для структурированных данных.
    *   **DataStore / SharedPreferences:** Для простых настроек и ключ-значение пар.
    *   **Файловая система:** Для сохранения файлов, изображений и т.д.
    *   **SavedStateHandle:** Специальный механизм в ViewModel для сохранения небольших объемов данных при смерти процесса (автоматически сериализует их в Bundle).
